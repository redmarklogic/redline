# Advanced Skill Techniques

Reference for skill authors reviewing or creating skills in `.agents/skills/`.
Covers Claude Code advanced frontmatter and body features, known bugs, and Redline project conventions.

**Official docs:** https://code.claude.com/docs/en/skills

---

## Decision Table

| Technique | Use when | Do NOT use when | Known bugs / caveats |
|---|---|---|---|
| `!` dynamic injection | Skill needs runtime state (git diff, branch, test output, env vars) as grounding context before Claude reasons | Skill is pure instruction with no environmental dependency | `shell:` must match OS — set `shell: powershell` on Windows; can be disabled globally with `disableSkillShellExecution: true` |
| `context: fork` | Investigation task that is read-only and should not pollute main context (research, analysis, debugging sweep) | Skill needs `AskUserQuestion` mid-run; skill uses `$ARGUMENTS` (broken in fork); skill writes files back to main session | `$ARGUMENTS` broken in forked context (#34164); infinite recursion risk (#55592); MCP tools leak despite `allowed-tools` (#40104); `AskUserQuestion` breaks (#19751) |
| `disable-model-invocation: true` | Destructive or gated actions (git push, issue creation, deployment, bulk deletes) that must only run on explicit user invocation | Skills Claude should auto-invoke; skills called from other skills or subagents | Still consumes context tokens when listed (#41417); blocks subagent invocation (#43809) |
| `arguments` / named args | Skill needs caller-supplied data (issue #, branch name, component, topic) to be specific — without it the skill guesses | Skill works correctly without external input | Use quoted args for multi-word values; `$ARGUMENTS`, `$0`/`$1`, or named `$name` substitution all available |
| `agent: Explore` | Fork + read-only discovery wanting lean context (no CLAUDE.md, no git status) | Skill needs to write files or use MCP write tools | Only meaningful combined with `context: fork` |
| `paths:` | Skill should auto-activate only for a specific file type or domain (Python-only, QMD-only, Markdown-only) | General-purpose skill applicable to any file | Glob pattern format; does not restrict manual invocation |
| `allowed-tools:` | Pre-approving specific tools to reduce permission prompts during skill execution | Trying to RESTRICT tool access — it grants pre-approval, it does NOT restrict | Does not block access (#37683); use scoped Bash filters: `Bash(git add *)` not `Bash(*)` |
| `disallowed-tools:` | Skill should block Claude from using a specific tool (e.g., block Write during a read-only audit) | General restriction (prefer `allowed-tools` positive list) | Restriction clears after the user's next message |
| `model:` override | Skill consistently needs a different capability tier (haiku for trivial, opus for heavy reasoning) | Cost saving alone — use `effort:` instead | Override applies to current turn only; not persisted |
| `effort:` override | Skill consistently benefits from a specific reasoning depth | General use without a strong signal | Available levels: `low`, `medium`, `high`, `xhigh`, `max` |
| Skill-scoped `hooks` | Deterministic pre/post enforcement tied to one specific skill (lint gate before implement, audit log after handover) | Plugin-hosted skills | Not triggered in plugins (#17688); define under `hooks:` in frontmatter |

## Project Conventions

1. **Vendor skills are read-only.** Skills prefixed `speckit-*` are generated by the `specify` CLI.
   Never apply any of the above techniques to them — `specify upgrade` will overwrite manual changes.

2. **`context: fork` requires an explicit return contract.** If a skill forks, its output section
   must describe what the subagent returns and how it surfaces to the user. A fork with no output
   contract is silent — the result is lost.

3. **`disable-model-invocation: true` must be documented inline.** Add a comment in the skill body
   explaining why it is manual-only (e.g., "Manual-only: pushes to remote — do not auto-invoke").

4. **`!` injection replaces runtime discovery instructions.** If a skill currently tells Claude to
   "check the current git diff" or "read the current branch name", replace those instructions with
   `!` injection and delete the discovery instruction. The agent starts from facts, not guesses.

5. **`paths:` is required for domain-specific skills.** All `python-*`, `qmd-*`, and file-format
   skills must declare `paths:` so they activate contextually rather than loading globally.

6. **`arguments` over vague prompts.** If a skill body says "replace X with the actual issue number"
   or "substitute the branch name here", that is a signal to use `arguments` with named substitution.

## Audit Checklist — Evaluating an Existing Skill

Work through this for each skill under review. Stop at the first "Vendor?" check.

- [ ] **Vendor?** Does the skill name start with `speckit-`? → Stop. Do not modify.
- [ ] **Runtime state?** Does the skill instruct Claude to discover environmental state (git status,
  current branch, test results, env vars, open PRs) at runtime? → Replace with `!` injection.
- [ ] **Read-only investigation?** Does the skill perform a bounded, read-only investigation
  (research, analysis, debugging sweep) with no user questions mid-run and no writes? → Candidate for `context: fork`.
- [ ] **Destructive/gated?** Does the skill trigger irreversible actions (push to remote, create
  GitHub issues, deploy, bulk delete)? → Set `disable-model-invocation: true` and add a manual-only note.
- [ ] **Needs caller input?** Does the skill body contain placeholders or tell Claude to "infer"
  what to operate on (issue number, branch, component, topic)? → Add `arguments` with named substitution.
- [ ] **Domain-specific?** Does the skill only apply to a specific file type or language?
  → Add `paths:` glob.
- [ ] **Permission-heavy?** Does the skill trigger many tool-permission prompts?
  → Add `allowed-tools:` for the specific safe tools it uses.
