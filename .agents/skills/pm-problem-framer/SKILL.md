# Problem Framer

## Overview

Force clarity on product problems before solution work begins. Every problem must name a user segment, a measurable outcome, and a strategic link — or it is not ready.

## When to Use

- Before writing an initiative or scoping a feature
- When a problem statement feels fuzzy or contested
- When stakeholders disagree on what the problem actually is

## Quick Reference

| Check | Pass | Fail |
|---|---|---|
| Target user | Named segment | "All users" or missing |
| Core pain | Specific, observable | Vague symptom |
| Desired outcome | Measurable metric | "Better", "faster" |
| Strategic link | Linked to bet | No connection |
| Constraints | Stated | Assumed or missing |

## Behaviour Rules

- Require a target user segment — reject "all users"
- Require a measurable desired outcome — reject outcome-free framing
- Flag any problem not linked to a strategic bet
- Ask at most 3 sharpening questions per response — prioritize the most critical gap
- Do not suggest solutions — this skill frames problems, not fixes

## Input

Provide a raw problem statement (1-3 sentences), a draft initiative, or a feature request. Optionally include the strategic bet, target segment, or known constraints.

## Output Structure

**Reframed Problem:** Cleaner, more precise version.

**Clarity Check:** Target user, Core pain, Desired outcome, Strategic link, Constraints — each rated Identified / Vague / Missing (or Measurable / Unmeasurable for outcome).

**Sharpening Questions:** Up to 3, most critical first.

**Structural Gaps:** Specific, actionable gaps. Omit if none detected.

## Common Mistakes

- **Accepting "all users"** — forces vague solutions that satisfy nobody.
- **Letting solutions creep in** — "we need a dashboard" is a solution, not a problem.
- **Framing symptoms not causes** — "reports are slow" describes a symptom; the problem is why.
- **Skipping the strategic link** — unlinked problems produce orphan initiatives.
