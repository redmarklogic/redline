## Engineering Lens Findings

### 4a. Architecture Soundness

The architecture is sound: `for_each` on environments, shared `locals` for secret mapping,
per-secret IAM members, v2 resource with explicit probes. Separation of concerns across
`cloud_run.tf`, `secrets.tf`, `iam.tf` is clean. No circular dependencies.

No findings.

### 4b. Failure Mode Analysis

Failure modes are mostly well-covered. Cloud Run handles unhealthy revisions by keeping
the prior revision active (SC-006). Startup probe gates initial traffic. Secret rotation
between deploys is acknowledged as a known idempotency deviation.

One gap: **E2** (💡 Recommendation) — no liveness probe is defined. The plan explicitly
decided to use only `startup_probe`. This is valid for readiness gating, but it means a
service that becomes unhealthy after startup (e.g., database connection pool exhausted)
will continue receiving traffic indefinitely until manually restarted or redeployed. Given
this is a production service, the absence of a `liveness_probe` is a meaningful operational
gap. version-guard-report.md rule #3 confirms `liveness_probe` is supported in 5.x.

### 4c. Security & Privacy Review

**E1** (🎯 Must-Address): The `google_secret_manager_secret_version` placeholder strategy
stores `secret_data = "REPLACE_ME"` in Terraform state. Terraform state is stored in GCS
and contains all resource attribute values in plaintext by default. While `"REPLACE_ME"` is
not a real secret, the same Terraform resource block — with `secret_data` as a tracked
attribute — will also record the *actual* secret value if Brent ever runs `terraform import`
on a live version or if the resource is inadvertently refreshed after rotation. The SC-002
checkov gate scans HCL source, not state. A state file containing a rotated real value would
pass checkov but violate FR-002.

Suggested resolution: Switch to a data-only pattern for actual secrets — Terraform declares
the `google_secret_manager_secret` resource (name, replication policy) but does NOT manage
`google_secret_manager_secret_version` at all. Actual versions are created out-of-band by
Brent via `gcloud secrets versions add`. This keeps secret values entirely out of Terraform
state. A `terraform_data` resource (or a `null_resource` with a `local-exec` that emits a
reminder, not a value) can document the manual step. Alternatively, mark the state bucket
as customer-managed encryption key (CMEK) encrypted and access-controlled — but eliminating
the tracked attribute is cleaner.

**IAM**: Per-secret IAM members (least-privilege) — correct. Single SA across environments
is a known accepted risk (research.md Decision 4). No additional findings.

**Ingress**: `INGRESS_TRAFFIC_ALL` allows unauthenticated public access. This is presumably
intentional for a backend API, but it is not stated in the spec. There is no requirement for
Cloud Run invoker IAM control. This is acceptable for an API with its own auth layer, but
worth confirming.

**E3** (💡 Recommendation): checkov is run against the HCL source directory, not the plan
output (`tfplan.json`). `checkov -d` scans static files; it cannot detect values that are
only resolved at plan time (e.g., a `local.secret_value` derived from a data source).
The SC-002 gate would be more robust if it scanned the JSON plan output:
`checkov --file tfplan.json --framework terraform_plan`. Both should be run.

### 4d. Performance & Scalability

Max-instances defaults (3 for staging, 10 for prod) are documented in the plan but their
rationale is not stated. This is acceptable for an early deploy — the spec notes cost
protection is the goal. No performance bottlenecks in the Terraform design itself.

No findings.

### 4e. Testing Strategy

`terraform validate` + `terraform plan` (no apply in PR) is appropriate for IaC. checkov
gate is correct. Manual smoke tests are the right fit for a first deploy with no CI.
Idempotency test (Step 10) is well-specified.

No findings.

### 4f. Operational Readiness

Rollback strategy (SC-006) is built into Cloud Run's traffic management — correct. No
manual migration required. No database migrations. Logging/metrics come from Cloud Run
and GCP by default — not explicitly called out but implied.

One gap already captured as E2 (liveness probe).

The quickstart.md Step 10 uses `terraform apply -auto-approve` for the idempotency test.
This should be flagged (captured as P1 above).

### 4g. Dependencies & Integration Risks

All external dependencies (Artifact Registry, Secret Manager, Cloud Run) are GCP-managed.
The pre-flight `gcloud artifacts docker images describe` check (F-RT-70-005) is correctly
implemented as a blocking step in quickstart.md. No third-party SaaS dependencies. No
API versioning concerns — single consumer (Brent + Terraform).

No additional findings.

## Verdict

⚠️ **PROCEED WITH UPDATES**

E1 is a must-address: the `google_secret_manager_secret_version` resource in Terraform creates
a tracked state attribute that will eventually hold a real secret value if the workflow evolves
(import, refresh, or Brent forgets to disable the Terraform-managed version before rotating).
This directly violates FR-002.

E2, E3, P1 are recommendations that strengthen the spec without blocking.

---

## Remediation Proposals

### E1 — Remove `google_secret_manager_secret_version` from Terraform (spec.md + plan.md)

**plan.md change** — Phase 0 Resolved Decisions, Decision 6:

> **Current**: "Secret version stubs use `google_secret_manager_secret_version` with `secret_data` set to a placeholder..."
>
> **Proposed**: "Terraform declares `google_secret_manager_secret` resources only (name, replication, labels). No `google_secret_manager_secret_version` resource is managed by Terraform. Initial secret values and all subsequent rotations are performed out-of-band by Brent using `gcloud secrets versions add`. This keeps all secret values entirely absent from Terraform state."

**plan.md change** — Phase 1 Data Model (remove `Secret Version (stub)` row):

> Remove row: `Secret Version (stub) | google_secret_manager_secret_version | 1 per secret (placeholder)`

**quickstart.md change** — Step 5 note:

> Add explicit note: "Terraform does NOT create secret versions. Step 5 (below) must be completed before any Cloud Run deploy — the service will fail startup if no enabled secret version exists."

### E2 — Liveness probe decision (plan.md)

**plan.md change** — Phase 1 Health-Check Probe Configuration:

> Add: "**Liveness probe**: Not configured in this feature. Rationale: Cloud Run scales to zero (staging) and is expected to be redeployed on failure rather than restarted in-place (prod). Accepted risk: a degraded-but-not-crashed instance will continue receiving traffic. Revisit when uptime SLA is formalised."

This converts an implicit gap to an explicit accepted-risk note — no behavior change, but removes ambiguity.

### E3 — checkov plan-output scan (quickstart.md + plan.md)

**quickstart.md Step 3** addition:
```bash
# Also scan the plan JSON for runtime-resolved values
checkov --file tfplan.json --framework terraform_plan --compact
```

**plan.md SC-002 Verification Gate** addition:
> "Run checkov against both the HCL source directory (`-d`) and the JSON plan output (`--framework terraform_plan`) for full coverage."

### P1 — Remove `-auto-approve` from quickstart.md Step 10

**quickstart.md Step 10** replacement:
```bash
# Use plan to verify idempotency without applying
terraform plan -var="image_tag=<SAME_TAG>" -detailed-exitcode
# Exit code 0 = no changes (idempotent). Exit code 2 = changes detected (investigate).
```
