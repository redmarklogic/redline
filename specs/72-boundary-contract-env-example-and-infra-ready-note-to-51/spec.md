# Feature Specification: Boundary Contract — .env.example and Infra-Ready Note

**Feature Branch**: `feature/72-boundary-contract-env-example-and-infra-ready-note-to-51`

**Created**: 2026-06-12

**Status**: Draft

**Input**: GitHub issue #72 — "Boundary contract: .env.example and infra-ready note to #51"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Developer reads .env.example and knows exactly what to set (Priority: P1)

A developer cloning the repository for the first time opens `.env.example`. They can
read every variable required to connect the application to the live Cloud Run
infrastructure: the API base URL, the OIDC token audience for service-to-service
calls, and the Cloud Run service account email. Each variable has a comment
explaining what it is, where to obtain the value, and whether it is a secret.

**Why this priority**: Without this, the app team (issue #51) cannot wire up their
environment and the entire deploy chain stalls. This is the primary deliverable of the
issue.

**Independent Test**: Set only the three new variables in a local `.env`, point the
app at staging, and confirm the connection works without any additional configuration
hunting.

**Acceptance Scenarios**:

1. **Given** a clean repo clone, **When** a developer reads `.env.example`, **Then**
   they see annotated entries for `API_BASE_URL`, `GCP_OIDC_AUDIENCE`, and
   `CLOUD_RUN_SA_EMAIL` with source instructions for each.

2. **Given** the existing `API_BASE_URL` entry (added per ADR-027), **When** the
   spec is implemented, **Then** the entry is updated or confirmed correct — no
   duplicate variable exists.

3. **Given** the variables are classified correctly, **When** a developer reviews
   `.env.example`, **Then** secrets (values that must not be committed) are marked
   `SECRET` and non-secrets are marked `Non-secret`.

---

### User Story 2 — Issue #51 receives an infra-ready notification (Priority: P2)

The app team working on issue #51 (Skeleton endpoint: POST /skeletons) needs to know
that infrastructure is provisioned and ready for them to connect. Brent posts a
structured comment on issue #51 confirming readiness, listing the variables and their
values (non-secrets only), and declaring any remaining gaps.

**Why this priority**: Issue #51 is blocked on this signal. Without it, the app team
cannot confidently start wiring the endpoint to the live infrastructure.

**Independent Test**: Navigate to issue #51 on GitHub; an infra-ready comment exists,
is structured, and contains at least the staging `API_BASE_URL` and the `CLOUD_RUN_SA_EMAIL`.

**Acceptance Scenarios**:

1. **Given** infra is provisioned, **When** this issue is closed, **Then** a comment
   on issue #51 exists with heading "Infra ready" and lists all three variables with
   their staging values (non-secrets inline, secrets noted as "set in Secret Manager").

2. **Given** the comment is posted, **When** the app team reads it, **Then** they can
   reproduce the values independently from the stated sources (no value requires
   Brent to be present).

---

### Edge Cases

- What if `API_BASE_URL` already in `.env.example` uses a stale Cloud Run hash URL?
  The spec-time value may differ from the deployed URL — the comment must instruct
  readers to read the live value from the deployed service, not trust the example.
- What if the Cloud Run service has not yet been deployed (blocker B7 not cleared)?
  The `.env.example` entries can be written with placeholder format values; the
  infra-ready comment on #51 must not be posted until B7 is cleared.
- What if the OIDC audience is the same string as `API_BASE_URL`? Document this
  explicitly rather than omitting the variable — the duplication must be intentional
  and visible.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `.env.example` MUST contain an annotated entry for `API_BASE_URL`
  (Cloud Run service URL for the target environment) that instructs readers to read
  the live value from the deployed service rather than hard-coding the URL.

- **FR-002**: `.env.example` MUST contain an annotated entry for `GCP_OIDC_AUDIENCE`
  (the token audience string an application uses when verifying inbound OIDC tokens
  issued by GCP — e.g., tokens generated by Cloud Run or Workload Identity callers),
  clearly noting it equals the Cloud Run service URL for Google-issued tokens.
  Note: at POC stage there are no outbound service-to-service calls (connection
  strategy is inbound-only); this variable is documented for future consumers.

- **FR-003**: `.env.example` MUST contain an annotated entry for `CLOUD_RUN_SA_EMAIL`
  (the email address of the Cloud Run service account, used by app-level code that
  needs to reference the SA for IAM or audit purposes), with the canonical value
  derivable from Terraform outputs.

- **FR-004**: Every new `.env.example` entry MUST be tagged `Non-secret` or `SECRET`
  in its inline comment, consistent with the annotation style of existing entries.

- **FR-005**: A comment MUST be posted on GitHub issue #51 with the heading "Infra
  ready" once B7 is cleared, listing: `API_BASE_URL` (staging value), `CLOUD_RUN_SA_EMAIL`
  (value), and `GCP_OIDC_AUDIENCE` (value or equivalence note), plus any known gaps
  or caveats.

- **FR-006**: The infra-ready comment MUST NOT contain raw secret values. Secret
  Manager secret names may be included; actual token or key values must not.

### Key Entities

- **`.env.example`**: Root-level template listing all environment variables required
  to run the application against live infrastructure. Non-secrets are shown with
  guidance on how to obtain the value; secrets are left blank with a note.
- **Infra-ready comment**: A structured GitHub comment on issue #51, acting as the
  formal hand-off signal from infra to the app team.
- **Cloud Run service account**: The GCP service account under which the Cloud Run
  container runs; its email is a non-secret identity value consumed by app code.
- **OIDC audience**: The string the application uses to verify that an incoming token
  was issued for this specific service; equals the Cloud Run service URL for
  Google-issued tokens.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A developer following only `.env.example` can configure a local
  environment that successfully connects to staging within one working session
  (no Slack/Brent required).

- **SC-002**: Issue #51 unblocks within one working day of this issue closing —
  the infra-ready comment provides all values the app team needs to complete their
  wiring.

- **SC-003**: Zero secret values appear in `.env.example` or the GitHub comment
  (verified by `detect-secrets` scan on the file and a manual review of the comment).

- **SC-004**: The `.env.example` change passes CI (detect-secrets baseline, no
  new findings flagged as genuine secrets).

## Assumptions

- B7 (the blocker) refers to a project board item that must be cleared before the
  infra-ready comment can be posted on #51; `.env.example` itself can be written
  regardless.
- The Cloud Run services (staging + prod) are already provisioned via `terraform apply`
  (spec-70); the service URLs and SA email are obtainable from `terraform output`.
- `API_BASE_URL` already exists in `.env.example` from ADR-027; this issue either
  confirms it is correct or updates its annotation to match the deploy-chain contract.
- IAP (Identity-Aware Proxy) is out of scope per ADR-022 and the cloud-run-connection-strategy
  document; no IAP client ID variable is required.
- The OIDC audience for Cloud Run service-to-service calls equals the Cloud Run
  service URL (Google-issued tokens use the service URL as audience by default).
