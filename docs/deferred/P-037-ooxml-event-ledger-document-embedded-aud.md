# OOXML Event Ledger — document-embedded audit trail

## Why deferred

*(Broadened 2026-05-09 from "skeleton acceptance rate / OOXML provenance tracking".)* Multiple use cases converge on the same OOXML infrastructure: (1) **Skeleton provenance** — was this document generated by Redline? How much was changed? (2) **Engineer signature tracking** — which engineers reviewed, approved, or signed off on the document, and when? (3) **Future events** — standards applied, Pre-Review annotations accepted/rejected, House Rules compliance state. Rather than bespoke markers for each, design a single extensible **event ledger** as a custom XML part embedded in every generated DOCX. The ledger is a structured, append-only log of typed events with timestamps and actor identifiers. **Decisions (2026-05-09):** Sprint 1 embeds the **provenance event only** (skeleton generation timestamp, template version, standards applied) — write-only, nothing reads it yet. **Full ledger schema design** (extensibility model, all event types, versioning rules) is a Sprint 2 engineering design task. Signature event type design and Graeme domain validation are deferred to when signature work actually begins. Phase 2 reads the ledger for acceptance-rate measurement and signature audit trail. Connects to the insurance bifurcation requirement (Bet 2): "Redline gives you the audit trail your insurer will ask for."

**Provenance event embedding**: Sprint 1 (ships with first DOCX). **Full ledger schema design**: Sprint 2. **Signature event type + Graeme validation**: deferred to when signature work begins. **Ledger reading**: Pre-Review engine is live (Phase 2).
